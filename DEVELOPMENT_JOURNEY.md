# **Development Journey & Thought Process: Local STT/ASR Electron App**

## **Project Origins & Core Objectives**

The original vision for this project was to explore and simplify **Local STT/ASR (Speech-to-Text / Automatic Speech Recognition) model deployment**. Relying on cloud APIs for transcription poses privacy, cost, and offline-availability concerns. The goal was to:

1. **Benchmark Open-Source Solutions:** Evaluate the performance and resource usage of models like OpenAI Whisper, Qwen3-ASR, and Voxtral Transcribe 2 in a realistic edge environment (e.g., Macbooks, Windows \+ GPU).  
2. **Build a GUI Prototype:** Create a user-friendly Electron.js desktop application that allows users to seamlessly test and interact with these benchmarked models.  
3. **Document the AI-Assisted Workflow:** Record the planning, prompting, reviewing, and testing processes using modern AI development tools.

## **Phase 1: Model Management & Auto-Installation**

The first major hurdle in local AI deployment is the environment. A user selecting a model in the frontend is meaningless if the binary or Python package isn't present on their system.

### **The Solution**

I prioritized building a robust check-and-install pipeline (src/core/installer.js and src/core/runtime.js). Before any transcription runs, the app verifies if the selected model is installed. If it is missing, the backend triggers an automated installation flow:

* Pulls weight files directly from Hugging Face.  
* Orchestrates pip install commands behind the scenes.  
* Validates executable binaries before attempting execution.

This transformed a previously fragile setup (which threw cryptic ENOENT command errors) into a smooth, self-healing process with real-time progress feedback on the frontend status bar.

## **Phase 2: Taming the AI Tools**

Throughout the project, I relied heavily on AI assistance, primarily using a **GitHub Copilot Agent** session to directly modify the repository, supplemented by **GitHub Copilot Chat**, **Claude Sonnet 4.6**, and **Gemini 3.1 Pro**.

### **The Challenge**

Initially, the code generated by the GitHub Copilot Agent running directly on the repository did not adhere to my preferred specifications and project architecture. It was introducing inconsistent patterns, missing documentation, and bypassing architectural rules.

### **The Fix**

I paused feature development to write explicit, strict instructions and coding guidelines for the AI models to follow. Establishing these strict guardrails drastically improved the quality and consistency of subsequent generations. Key rules included:

* Enforcing fs-extra instead of the native fs/promises.  
* Strict 80-character line length limits for backend logic.  
* Standardized camelCase naming conventions.  
* Comprehensive JSDoc documentation block requirements.

## **Phase 3: Voice Recording & The Encoding Bug**

To make the app truly useful, users needed to be able to record audio directly within the interface rather than just uploading pre-existing files. I implemented a built-in recording feature using the browser's native MediaRecorder API, adding a pulsing red dot indicator, real-time duration tracking, and automatic saving to an ./output/ directory.

### **The Hurdle**

While testing the recording functionality with my own voice in Chinese, I ran into a severe issue regarding ISO character encoding. The file paths and metadata were corrupted because the system encoding did not properly support the non-Latin Chinese character sets.

### **The Fix**

I stepped in and manually resolved the encoding logic in the pipeline, ensuring that files with non-Latin filenames or containing multi-byte characters are encoded and decoded correctly when passed between the Electron renderer process, the preload script, and the main Node.js process.

## **Phase 4: The Windows Environment & CUDA Headaches**

Because I am developing on a Windows machine without access to a Mac, ensuring hardware acceleration (GPU vs. CPU) became a primary focus—and a major technical bottleneck.

### **The Hurdle**

Many of these advanced STT models require CUDA to run efficiently on the GPU when on Windows. However, my machine had multiple versions of Python installed. The Windows py launcher was inadvertently grabbing the wrong Python environment—one without the proper PyTorch/CUDA bindings. As a result, heavy models were falling back to CPU execution, resulting in extremely slow transcription times.

### **The Fix**

I overhauled the command probing logic in src/core/runtime.js. The code now actively tests environment paths (py \-3, absolute paths, etc.) and allows for environment variables (like $env:PYTHON\_BIN) to ensure that the Electron app targets the exact Python executable configured with CUDA support.

## **Phase 5: Refactoring & Platform Stabilization**

The final stretch involved cleaning up the codebase to ensure it was maintainable and stable. I completed a massive refactor to enforce the AI guidelines established in Phase 2:

* Replaced all native file system promises with fs-extra.  
* Maintained strict 80-character line limits.  
* Added comprehensive JSDoc annotations to all backend modules.  
* Cleaned up CLI argument parsing across all adapters.

### **Current Platform Status**

Due to hardware constraints (lack of access to a macOS machine), I focused on stabilizing the application on Windows first. While macOS and Linux logic (like python3 command resolving and brew install whisper-cpp guidance) is fully written into the codebase, Windows is currently the only platform verified and thoroughly tested.

## **Phase 6: Next Steps & Future Roadmap**

To transition this prototype into a production-ready application, several key features, configurations, and build considerations should be integrated.

### **1\. High-Value Application Features**

* **Real-time Streaming Transcription:** Implement real-time microphone capturing paired with a lightweight streaming model (e.g., streaming whisper.cpp or Silero VAD) to display transcribed text as the user speaks, instead of waiting for the recording to finish.  
* **Voice Activity Detection (VAD):** Integrate a local VAD engine (such as WebRTC VAD or Silero VAD) to automatically pause recording during silences, trim dead air, and chunk long recordings into logical semantic segments.  
* **Speaker Diarization:** Implement speaker identification (e.g., using pyannote-audio or similar lightweight local pipelines) so the resulting transcripts show speaker turns (e.g., *Speaker 1: Hello*, *Speaker 2: Hi*).  
* **Local History and Database:** Add a persistent local database (such as SQLite, NeDB, or RxDB) to store past recordings, transcriptions, performance metadata (VRAM usage, speed, RTF), and custom labels.  
* **Advanced Subtitle & Format Exports:** Provide options to export transcriptions into .srt, .vtt, .json, .pdf, and .txt with customized timestamps and speaker formatting.

### **2\. Enhanced Configuration & Environment Tuning**

* **Quantization & Performance Profiles:** Allow users to choose their precision level in the UI (e.g., float16, int8, int4 quantization) to balance accuracy with VRAM usage, preventing out-of-memory (OOM) errors on lower-end GPUs.  
* **Unified Virtual Environment (venv) Management:** Instead of relying on the host's global Python environment—which is prone to version conflicts—the Electron app should automatically scaffold its own isolated Python virtual environment inside the application's user data directory on its first run.

### **3\. Production Packaging & Build Pipeline**

* **Multi-Platform Electron Builder Configuration:** Implement electron-builder with dedicated targets for Windows (nsis, portable), macOS (dmg, zip), and Linux (AppImage, deb).  
* **Code Signing and Notarization:** \* **Windows:** Secure a code-signing certificate (EV or standard) to bypass Microsoft SmartScreen warnings during installation.  
  * **macOS:** Integrate Apple Developer ID signing and the Electron Notarization toolchain (@electron/notarize) as part of the package hooks to ensure the app runs seamlessly on Gatekeeper.  
* **Native Dependencies Compilation:** Ensure that any native C++ Node bindings (such as local audio-capture libraries) are automatically compiled for the target platform using electron-rebuild during the package step.  
* **Automated CI/CD Workflows:** Create a GitHub Actions workflow that automatically spawns Windows, macOS, and Linux runners to build, sign, and draft releases whenever a new tag is pushed.

## **Toolchain Summary**

This project demonstrates a successful hybrid development workflow. While I designed the system architecture, established coding guidelines, and manually debugged complex environment bugs (like the Chinese character encoding and CUDA multi-python pathing), the heavy lifting of boilerplate, UI logic, and refactoring was accelerated by:

* **GitHub Copilot Agent:** For automated repository-wide modifications.  
* **GitHub Copilot Chat with Claude Sonnet 4.6 model:** For quick, context-aware inline queries, reasoning and refactoring.  
* **Gemini 3.1 Pro:** For alternative problem-solving and debugging advice.
